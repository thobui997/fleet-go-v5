<objective>
## Goal
Implement Supabase-based authentication with login/logout, session persistence, auth context, and protected routes — the security gate for all subsequent features.

## Purpose
Every feature module requires authenticated access. This plan establishes the auth layer: Supabase GoTrue integration, React context for session state, a login page with form validation, and route guards that redirect unauthenticated users.

## Output
- Auth context and hooks (useAuth, useSession) managing Supabase session state
- Login page with email/password form validation (React Hook Form + Zod)
- Protected route component guarding authenticated routes
- Router updated with login route and protected route wrappers
- Auth provider wired into app providers
</objective>

## AC-1: Auth Context Provides Session State
```gherkin
Given the Supabase client is configured
When the AuthProvider wraps the application
Then a useAuth hook is available returning { user, session, isLoading, isAuthenticated }
And a useSession hook is available returning the current Supabase session
And auth state persists across page refreshes via Supabase session management
And isLoading is true during initial session check and false after
```

## AC-2: Login Page Authenticates Users
```gherkin
Given the login page is rendered at /login
When a user submits valid email and password
Then Supabase signInWithPassword is called
And on success the user is redirected to the dashboard route
And on failure an error message is displayed
And the form validates email format and password presence using Zod
And the submit button shows a loading state during authentication
And the submit button is disabled during submission to prevent double-submit
<!-- audit-added --> And specific Supabase error codes are mapped to Vietnamese user-friendly messages
<!-- audit-added --> And during initial auth check (isLoading=true), the login page shows a loading state instead of the form or redirect
```

## AC-3: Protected Routes Guard Authenticated Access
```gherkin
Given a user is not authenticated
When they navigate to a protected route (e.g., /dashboard)
Then they are redirected to /login with the intended URL stored as a redirect parameter
And after successful login they are redirected to the originally intended URL
Given a user is authenticated
When they navigate to /login
Then they are redirected to /dashboard
```

## AC-4: Logout Clears Session
```gherkin
Given a user is authenticated
When they trigger logout
Then Supabase signOut is called
And the user is redirected to /login
And the session state is cleared
```

    1. Create `src/shared/auth/auth-context.tsx`:
       ```typescript
       Design requirements:
       - Create AuthContext with React.createContext
       - AuthProvider component that:
         a. Uses Supabase `supabase.auth.onAuthStateChange` listener to track session
         b. Calls `supabase.auth.getSession()` on mount for initial session check
         c. Stores user, session, isLoading state
         d. Provides login(email, password), logout(), signUp(email, password) functions
         e. Cleanup listener on unmount
       - AuthContextValue type:
         user: User | null
         session: Session | null
         isLoading: boolean
         isAuthenticated: boolean
         login: (email: string, password: string) => Promise<void>
         logout: () => Promise<void>
         signUp: (email: string, password: string) => Promise<void>
       ```

       <!-- audit-added: Subscription cleanup pattern -->
       **Auth state listener cleanup:** The `onAuthStateChange` call returns `{ data: { subscription } }`. Store this subscription and call `subscription.unsubscribe()` in the useEffect cleanup function. Without this, hot route changes and component remounts create duplicate listeners causing stale state updates and memory leaks.

       <!-- audit-added: Export AuthContextValue type -->
       **Type export:** Export the `AuthContextValue` interface from this file. Future features and pages need this type for type-safe auth consumption (e.g., `const { user } = useAuth()` should be fully typed). Add to the index.ts public API as `export type { AuthContextValue } from './auth-context'`.

    2. Create `src/shared/auth/use-auth.ts`:
       ```typescript
       - Custom hook that consumes AuthContext
       - Throws error if used outside AuthProvider
       - Returns the full AuthContextValue
       ```

    3. Create `src/shared/auth/use-session.ts`:
       ```typescript
       - Convenience hook that returns just the session object
       - Useful for components that only need session data (JWT tokens, etc.)
       ```

    4. Create `src/shared/auth/protected-route.tsx`:
       ```typescript
       - Component that checks authentication state via useAuth
       - Props: children (ReactNode)
       - If isLoading: render a centered loading spinner or Skeleton
       - If not authenticated: redirect to /login using Navigate from react-router-dom
         - Include current location in state: `state={{ from: location }}` for post-login redirect
       - If authenticated: render children
       - Uses useLocation() from react-router-dom for current path
       ```

    5. Create `src/shared/auth/index.ts` as public API:

       ```typescript
       export { AuthProvider } from './auth-context';
       ```

       ```typescript
       import { z } from 'zod';

       export const loginSchema = z.object({
         email: z.string().email('Vui lòng nhập email hợp lệ'),
         password: z.string().min(6, 'Mật khẩu phải có ít nhất 6 ký tự'),
       });

       export type LoginFormData = z.infer<typeof loginSchema>;
       ```
       Note: Vietnamese error messages for the target user base.

    2. Create `src/pages/login/ui/login-page.tsx`:
       ```typescript
       Design requirements:
       - Full-page centered layout with fleet branding
       - Card-based login form using Shadcn/ui Card component
       - App title/logo: "FleetGo" displayed prominently
       - Subtitle: "Hệ thống quản lý đội xe" (Fleet Management System)

       Form fields:
       - Email: Input with type="email", wrapped in FormFieldWrapper with label="Email" and required
       - Password: Input with type="password", wrapped in FormFieldWrapper with label="Mật khẩu" and required
       - Submit button: Button with "Đăng nhập" text, loading spinner when isSubmitting

       Behavior:
       - Use React Hook Form with zodResolver(loginSchema) for validation
       - On submit: call useAuth().login(email, password)
       - On success: navigate to location.state?.from or ROUTES.DASHBOARD
       - On error: display toast with error message using useToast
       - If already authenticated: redirect to dashboard

       <!-- audit-added: Auth error mapping -->
       **Error handling:** Map Supabase auth error codes to Vietnamese user-friendly messages. Create an `AUTH_ERROR_MESSAGES` map in login-page.tsx:
       - `invalid_credentials` → "Email hoặc mật khẩu không chính xác"
       - `email_not_confirmed` → "Tài khoản chưa được xác nhận. Vui lòng liên hệ quản trị viên."
       - `too_many_requests` → "Quá nhiều lần thử. Vui lòng thử lại sau."
       - Default/fallback → "Đã xảy ra lỗi. Vui lòng thử lại."
       The login function in auth-context should throw the Supabase error so the login page can read `error.message` or `error.code` for mapping.

       <!-- audit-added: Loading state during session check -->
       **Session loading guard:** Before checking "if already authenticated, redirect to dashboard," check `isLoading` first. While `isLoading` is true, render a centered Skeleton or spinner — do NOT redirect. This prevents the flash of login page for authenticated users who refresh or navigate directly.

       <!-- audit-added: Double-submit prevention -->
       **Double-submit prevention:** The submit button MUST be disabled while `isSubmitting` is true (from React Hook Form) OR while `isLoading` is true (from auth context). Use `disabled={isSubmitting || isLoading}` to prevent multiple concurrent authentication requests.

       Styling:
       - Full viewport height (min-h-screen), centered
       - Background: subtle gradient or muted background
       - Card with shadow, reasonable max-width (400px)
       - Use existing Shadcn/ui components (Card, Button, Input, FormFieldWrapper)
       ```
