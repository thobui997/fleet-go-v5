<objective>
## Goal
Apply the viewport-constrained layout pattern to all 11 list pages (only the table body scrolls) and add opt-in scroll to the 4 non-list pages that need full-page scrolling.

## Purpose
Plan 09-01 established the foundation (AppLayout + DataTable). This plan wires up every page to use it. List pages get a two-section flex layout (fixed chrome + scrollable DataTable). Non-list pages get `h-full overflow-y-auto` so their content doesn't get clipped by the now-hidden overflow on `<main>`.

## Output
- 11 list pages: restructured root layout → fixed header/filter section + `flex-1 min-h-0` table section
- 4 non-list pages: root div gains `h-full overflow-y-auto`
- Double `p-6` padding resolved: page roots drop their own `p-6` (main already provides it)
</objective>

## AC-1: List pages — only table body scrolls
```gherkin
Given any of the 11 list pages is open on a desktop viewport
When the page has enough rows to overflow the available height
Then the page header (title + action button) and filter bar are fully visible without scrolling
And only the table body scrolls vertically
And the pagination bar is always visible below the table
And no scrollbar appears on the browser window itself
```

## AC-2: List pages — sticky header visible while scrolling
```gherkin
Given a list page with 20+ rows is open
When the user scrolls the table body down
Then the column header row remains pinned at the top of the table area
And the page title/filters above remain visible (they do not scroll away)
```

## AC-3: List pages — empty and error states fill available space correctly
```gherkin
Given a list page where the query returns an error or zero results
When the error panel or empty-state message is rendered instead of DataTable
Then it occupies the table section area without pushing pagination below the viewport
And the page header and filters remain visible above
```

## AC-4: List pages — mobile unaffected
```gherkin
Given any list page is open on a mobile viewport (<768px)
When the user interacts with the page
Then normal full-page scrolling applies — no sticky enforcement, no clipping
```

## AC-5: Non-list pages — content scrolls within viewport
```gherkin
Given the Dashboard, My Schedule, Trip Calendar, or Check-in page is open
When the content is taller than the viewport
Then the content scrolls within the <main> area (not the browser window)
And the sidebar and header remain visible at all times
```

## AC-6: No double padding
```gherkin
Given any page is open
When inspecting the computed padding around the page content in DevTools
Then the content has exactly one layer of p-6 (24px) padding — from <main>, not from the page root div
```

  <action>
    Apply the following structural pattern to every list page. Read each file before editing.
    The exact inner content (columns, filters, dialogs) is different per page — only the wrapper structure changes.

    **Pattern to apply:**

    BEFORE (current structure on all list pages):
    ```tsx
    return (
      <div className="space-y-6 p-6">          // ← ROOT: remove p-6 and space-y-6
        <div className="flex items-center justify-between">  // page title + action btn
          ...
        </div>
        <div className="flex items-center gap-4">             // filter bar
          ...
        </div>
        {isError ? <ErrorPanel /> : <DataTable ... />}
        <FormDialog ... />
        <DeleteDialog ... />
      </div>
    );
    ```

    AFTER (target structure):
    ```tsx
    return (
      <div className="flex flex-col h-full">   // ← ROOT: flex column, fills main's content area
        <div className="flex-none space-y-4 pb-4">  // ← FIXED CHROME: header + filters, does not scroll
          <div className="flex items-center justify-between">  // page title + action btn
            ...
          </div>
          <div className="flex items-center gap-4">           // filter bar
            ...
          </div>
        </div>
        <div className="flex-1 min-h-0">       // ← TABLE SECTION: fills remaining height, allows shrink
          {isError ? <ErrorPanel /> : <DataTable ... />}
        </div>
        <FormDialog ... />                     // ← dialogs stay outside layout divs (unchanged)
        <DeleteDialog ... />
      </div>
    );
    ```

    **Key rules:**
    - Remove `p-6` from the root div — `<main>` in AppLayout already provides it (fixing existing double-padding)
    - Remove `space-y-6` from the root div — flex + gap handles spacing
    - The fixed chrome wrapper gets `flex-none space-y-4 pb-4` — `pb-4` provides gap between filters and table
    - The table section wrapper gets `flex-1 min-h-0` — `min-h-0` is CRITICAL; without it flex children won't shrink below content height, defeating the scroll constraint
    - DataTable already has `h-full flex flex-col` (from 09-01) — it fills `flex-1 min-h-0` correctly
    - Dialogs (FormDialog, DeleteDialog, etc.) are portaled — they must remain OUTSIDE the flex layout divs, at the bottom of the return statement (unchanged from current position)

    **Per-page variations to watch for:**
    - Some pages have a multi-row filter bar (e.g., Maintenance has vehicle + type filter + date range) — keep all filters inside the `flex-none` section
    - Trips page has a tab bar or additional controls — include all non-table chrome in `flex-none`
    - Bookings page: the error state is a separate block — it goes inside the `flex-1 min-h-0` div alongside DataTable
    - Payments page: same structure as other list pages
    - If a page has an inline error banner (not full-page error) above the table, keep it in `flex-none`

    **Inline action dropdowns (DropdownMenu in table rows):**
    Radix UI `DropdownMenuContent` is portaled to `document.body` and uses `z-50`. The DataTable's
    sticky `<TableHeader>` uses `z-10`. No conflict — confirm this does NOT need adjustment.
    If any page uses an inline non-portaled Select inside a table cell (not standard), flag it in the summary.

    After editing all 11 files, run `npm run build` to confirm zero TypeScript errors before proceeding.
