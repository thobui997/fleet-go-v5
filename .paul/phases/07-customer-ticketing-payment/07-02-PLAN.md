<objective>
## Goal
Create the booking management module: booking + ticket entity slices, booking list page with filters, booking creation dialog with passenger/seat entry, booking detail view, booking cancellation with payment sync, and delete with error handling.

## Purpose
Bookings are the central revenue-generating entity in FleetGo — they connect customers to trips via tickets and track payment status. This plan delivers the full booking management workflow, enabling ticketing agents to create, view, and cancel bookings.

## Output
- Booking entity slice (types, API, TanStack Query hooks)
- Ticket entity slice (types, API, TanStack Query hooks)
- Booking CRUD UI (list, create, detail, cancel, delete)
- Router integration replacing bookings placeholder
</objective>

## AC-1: Booking List Page
```gherkin
Given the user navigates to /bookings
When the page loads
Then a paginated DataTable displays bookings with columns: Mã đặt vé, Khách hàng, Chuyến (route origin→destination), Ngày đặt, Tổng tiền, Trạng thái, Số KH
And a status filter dropdown (all/pending/confirmed/cancelled/completed/refunded) is available
And a date range filter (from/to) is available
And search by booking code or customer name works with 300ms debounce
And bookings are ordered by booking_date descending
```

## AC-2: Booking List Error State
```gherkin
Given the booking list query fails (network error, auth expiry)
When the error occurs
Then an error state is shown with AlertCircle icon, Vietnamese error message (mapSupabaseError handles 401/403/PGRST301), and a retry button
```

## AC-3: Create Booking
```gherkin
Given the user clicks "Tạo đặt vé" button
When the creation dialog opens
Then the user can:
  1. Select a customer (search dropdown with customer name/phone)
  2. Select a trip (dropdown showing route name, departure time, vehicle)
  3. Add passengers: for each, enter passenger name and seat number
  4. Remove a passenger row
  5. See total amount auto-calculated from ticket prices
When submitted with valid data
Then booking is created with status 'pending', booking_code auto-generated, and all tickets created
And the list refreshes with the new booking
```

## AC-4: Booking Detail View
```gherkin
Given the user clicks a booking row in the list
When the detail dialog opens
Then it shows: booking code, customer info, trip info (route, departure time, vehicle), booking date, status, total amount, notes
And a table of tickets (seat, passenger name, price, status)
And payment status summary if payment exists (method, amount, status)
And action buttons based on status (Cancel for pending/confirmed)
```

## AC-5: Cancel Booking
```gherkin
Given a booking with status 'pending' or 'confirmed'
When the user clicks "Hủy đặt vé" and confirms
Then booking status updates to 'cancelled'
And all active tickets update to 'cancelled' (releasing seats for the partial unique index)
And if a payment exists with status 'completed', payment status updates to 'refunded' with refunded_at = now()
And if a payment exists with status 'pending', payment status updates to 'failed' (not refunded — nothing was paid)
And the booking list refreshes
```

## AC-6: Vietnamese Error Handling
```gherkin
Given any booking operation encounters an error
When the error is displayed to the user
Then mapBookingError maps SQLSTATE codes to Vietnamese messages:
  - 23505 on idx_tickets_no_double_booking → "Ghế [seat] trên chuyến này đã được đặt"
  - 23505 on bookings_booking_code_key → should not happen (auto-generated), generic duplicate message
  - 23503 → "Không thể xóa đặt vé đã có thanh toán"
  - Auth expiry (401/403/PGRST301) → session expired message
  - Validation errors → Vietnamese Zod messages
```

## AC-7: Router Integration
```gherkin
Given the application loads
When navigating to /bookings
Then BookingsPage renders (not PlaceholderPage)
And all other routes remain unchanged
```

Join customer (full_name, phone_number), trip→route (name, origin_station→name, destination_station→name), trip→vehicle (license_plate). Filters: status eq, booking_date gte/lte, search .or() on booking_code ilike and customer.full_name ilike. Trim search before ilike. Use nested PostgREST FK join syntax.
       - fetchBooking(id): SELECT single with same joins + tickets array (id, seat_number, passenger_name, price, status)
       - createBookingWithTickets(input: { booking: BookingInsert, tickets: TicketInsert[] }): Sequential insert with compensating transaction. (1) Insert booking first (get booking_code from response). (2) Insert tickets with booking_id. (3) If ticket insert fails, DELETE the just-created booking (compensating transaction) to prevent orphaned bookings, then throw the original error. Calculate total_amount = sum of ticket prices, passenger_count = tickets.length.
       - cancelBooking(id): Update booking status='cancelled', cancelled_at=now(). Then update tickets status='cancelled' WHERE booking_id=id AND status IN ('active'). Then update payment: status='refunded', refunded_at=now() WHERE booking_id=id AND status='completed'; status='failed' WHERE booking_id=id AND status='pending'. <!-- audit-added: pending→'failed' not 'refunded' — can't refund what wasn't paid -->
       - deleteBooking(id): Delete booking (CASCADE deletes tickets, RESTRICT if payment exists).

    3. `api/booking.queries.ts`:
       - useBookings(params) — query with ['bookings', params] key
       - useBooking(id) — query with ['booking', id] key
       - useCreateBooking — mutation, invalidate ['bookings'] on success
       - useCancelBooking — mutation, invalidate ['bookings'] and ['booking', id] on success
       - useDeleteBooking — mutation, invalidate ['bookings'] on success

    4. `index.ts`: Export types, hooks, BOOKING_STATUSES
