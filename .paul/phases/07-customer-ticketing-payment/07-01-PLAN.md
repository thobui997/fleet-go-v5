```gherkin

Then a paginated table displays customers ordered by full_name
And each row shows full_name, phone_number, email, id_card_number, gender, created_at
And a search input filters by full_name, phone_number, or email (debounced 300ms)
And pagination controls allow page size selection (10/20/50/100)
```


## AC-3: Create Customer
```gherkin
Given a user with 'customers:write' permission clicks the create button
When a form dialog opens with fields: full_name (required), phone_number (required), email, date_of_birth, gender, id_card_number, address, notes
And the user fills in valid data and submits
Then the customer is created and appears in the list
And a success toast is shown
```

## AC-4: Edit Customer
```gherkin
Given a user with 'customers:write' permission clicks edit on a customer row
When the form dialog opens pre-filled with customer data
And the user modifies fields and submits
Then the customer record is updated in the list
And a success toast is shown
```

## AC-5: Delete Customer
```gherkin
Given a user clicks delete on a customer row
When the confirmation dialog appears
And the user confirms deletion
Then if the customer has no bookings, the customer is removed from the list
And if the customer has bookings (23503 FK violation), a Vietnamese error message is shown: "Không thể xóa khách hàng đã có đặt vé"
```

## AC-6: Validation Errors
```gherkin
Given a user submits the customer form with invalid data
When full_name is empty, phone_number is empty or invalid format, email format is invalid, or date_of_birth is in the future
Then Vietnamese validation messages are shown for each invalid field
And the form does not submit
And phone_number shows "Số điện thoại không hợp lệ (10-11 số, bắt đầu bằng 0)" for non-matching input
And date_of_birth shows "Ngày sinh không được là ngày trong tương lai" for future dates
```

## AC-7: Duplicate Constraint Errors


    ```typescript
    export interface Customer {
      id: string;
      full_name: string;
      phone_number: string;
      email: string | null;
      date_of_birth: string | null;  // ISO date string
      gender: string | null;         // 'male' | 'female' | 'other' | null

      address: string | null;
      loyalty_points: number;        // read-only, no UI in this phase

      created_at: string;
      updated_at: string;
    }

    export type CustomerInsert = Omit<Customer, 'id' | 'created_at' | 'updated_at'>;
    export type CustomerUpdate = Partial<CustomerInsert>;

    export interface CustomerListParams {
      page: number;
      pageSize: number;
      search?: string;  // ilike on full_name, phone_number, or email via .or()
    }
    ```

    **customer.api.ts:**
    - Follow station.api.ts pattern exactly
    - `fetchCustomers(params)`: select all columns, paginate with .range(), search with `.or('full_name.ilike.%{q}%,phone_number.ilike.%{q}%,email.ilike.%{q}%')` where q is `(params.search ?? '').trim()`, order by full_name ascending <!-- audit-added: search trim before ilike -->

    - Invalidate ['customers'] on mutations

    **index.ts:**
    - Export types: Customer, CustomerInsert, CustomerUpdate, CustomerListParams
    - Export hooks: useCustomers, useCustomer, useCreateCustomer, useUpdateCustomer, useDeleteCustomer

    Create the customer page layer following the stations page pattern exactly.

    **customer-form-schema.ts:**
    - Zod schema with Vietnamese error messages:
      - full_name: z.string().trim().min(1, 'Họ tên không được để trống')
      - phone_number: z.string().trim().min(1, 'Số điện thoại không được để trống').regex(/^(0\d{9,10})$/, 'Số điện thoại không hợp lệ (10-11 số, bắt đầu bằng 0)') <!-- audit-added: phone format regex -->
      - email: z.string().email('Email không hợp lệ').optional().or(z.literal(''))
      - date_of_birth: z.string().optional().or(z.literal('')).refine(val => !val || val <= new Date().toISOString().slice(0, 10), 'Ngày sinh không được là ngày trong tương lai') — date input returns string or empty <!-- audit-added: future date guard -->
      - gender: z.string().optional().or(z.literal('')) — Select component with options: Nam, Nữ, Khác mapping to 'male', 'female', 'other'
      - id_card_number: z.string().trim().optional().or(z.literal(''))
      - address: z.string().optional().or(z.literal(''))
      - notes: z.string().optional().or(z.literal(''))
    - `serializeToInsert(values)`: convert empty strings to null for nullable fields. **CRITICAL for email and id_card_number** — both are UNIQUE + nullable; PostgreSQL allows multiple NULLs but rejects multiple empty strings with 23505. Failing to coerce '' → null will cause false duplicate errors when creating multiple customers without email or id_card_number. Additionally: phone_number must be trimmed (prevents " 0901..." vs "0901..." UNIQUE bypass). id_card_number must be trimmed. Handle date_of_birth: empty → null. Exclude loyalty_points (server default 0). Gender Select uses DB values directly ('male'/'female'/'other'), so no mapping needed in serializeToInsert — just pass through or null for empty. <!-- audit-added: phone_number trim, email/id_card_number null coercion criticality, gender mapping clarification -->
    - `mapSupabaseError(error)`:
      - Auth: 401/403/PGRST301 → session expired message
      - 23505: discriminate by constraint:
        - message includes 'customers_phone_number_key' → "Số điện thoại đã tồn tại"
        - message includes 'customers_email_key' → "Email đã tồn tại"
        - message includes 'customers_id_card_number_key' → "Số CMND/CCCD đã tồn tại"
        - fallback → "Giá trị đã tồn tại"
      - 23503 → "Không thể xóa khách hàng đã có đặt vé"
      - 23514 → CHECK constraint violation message
      - default → generic failure message

      - full_name: Input (required)

      - gender: Select with placeholder "Chọn giới tính", options using DB values directly: {value: 'male', label: 'Nam'}, {value: 'female', label: 'Nữ'}, {value: 'other', label: 'Khác'} — serializeToInsert just passes through or null for empty, no label→value mapping needed <!-- audit-added: gender uses DB values in Select -->
      - id_card_number: Input
      - address: Textarea (or Input)
      - notes: Textarea
    - Use the Radix Select __none__ sentinel pattern for nullable Select fields
