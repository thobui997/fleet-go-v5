<acceptance_criteria>

## AC-1: Payment List Page
```gherkin
Given a user navigates to /payments
When the page loads
Then a paginated table displays all payments with columns: booking code, customer name, amount, method, status, paid date
And filters are available for: status (pending/completed/failed/refunded), method (cash/e_wallet/bank_transfer), date range
And search works by booking code
And the table supports pagination (10/20/30 per page)
```

## AC-2: Payment Status Update
```gherkin
Given a payment with status 'pending'
When the user clicks the status update action
Then a dialog shows payment details and allows updating status to 'completed' or 'failed'
And marking as 'completed' sets paid_at to current timestamp and processed_by to current user
And marking as 'failed' does not set paid_at and sets processed_by to current user
And illegal transitions (e.g. failed→completed, refunded→anything) are rejected with Vietnamese error
And the payment list refreshes after update
```

## AC-3: Payment Refund
```gherkin
Given a payment with status 'completed'
When the user clicks the refund action
Then a confirmation dialog appears
And upon confirmation, status changes to 'refunded' and refunded_at is set to current timestamp
And processed_by is set to current user
And an optional notes field captures the refund reason
```

## AC-4: Booking Detail Payment Integration
```gherkin
Given a user opens a booking detail dialog
When the booking has an associated payment
Then the payment section shows: method, status badge, amount, paid_at date, transaction reference, and refund info if applicable
And if no payment exists, shows "Chưa có thanh toán" (existing behavior)
```

## AC-5: Error Handling
```gherkin
Given any payment API call fails
When the error is a constraint violation (23505, 23514) or auth error (401, 403, PGRST301)
Then a Vietnamese error message is displayed matching the error type
And the list page shows an error state with retry button
```

## AC-6: List Error State
```gherkin
Given the payment list fetch fails
When an error occurs
Then an error display appears with AlertCircle icon, Vietnamese error message, and retry button
And retry button calls refetch()
```

    **types.ts** — Define:
    - `PaymentStatus = 'pending' | 'completed' | 'failed' | 'refunded'`
    - `PaymentMethod = 'cash' | 'e_wallet' | 'bank_transfer'`
    - `PAYMENT_STATUSES` const array (like BOOKING_STATUSES pattern)
    - `PAYMENT_METHODS` const array
    - `Payment` interface matching DB schema: id, booking_id, amount, method, status, transaction_reference, paid_at, refunded_at, processed_by, notes, created_at, updated_at
    - `PaymentWithDetails extends Payment` — includes nested: `booking: { id, booking_code, customer: { id, full_name, phone_number } }` and optional `processed_by_profile: { id, full_name } | null`
    - `PaymentListParams` — page, pageSize, status?, method?, dateFrom?, dateTo?, search? (for booking_code ilike)

    **payment.api.ts** — Implement:
    - `PAYMENT_SELECT` constant with nested booking+customer join (use FK constraint name pattern `!payments_booking_id_fkey` to disambiguate if needed)
    - `fetchPayments(params: PaymentListParams)` — paginated list with filters (status, method, dateFrom on created_at, dateTo on created_at, search on booking_code via ilike — **trim search before ilike**). Order by created_at desc. <!-- audit-added: changed date filter from paid_at to created_at (pending payments have null paid_at and would be excluded); added search trim -->
    - `fetchPaymentByBooking(bookingId: string)` — single payment for booking detail view. Use `.eq('booking_id', bookingId).maybeSingle()` (not `.single()`) since booking may not have payment yet.
    - `UpdatePaymentStatusInput` type: `{ status: PaymentStatus; notes?: string; processed_by?: string }`
    - `updatePaymentStatus(id: string, input: UpdatePaymentStatusInput)` — validate transition + update with conditional timestamp logic:
      - **Validate allowed transitions:** pending→(completed|failed), completed→refunded. Throw Error with Vietnamese message for illegal transitions (e.g., "Không thể chuyển trạng thái từ {current} sang {target}")
      - status='completed' → also set `paid_at: new Date().toISOString()`, set `processed_by` from input
      - status='failed' → do NOT set paid_at, set `processed_by` from input
      - status='refunded' → also set `refunded_at: new Date().toISOString()`, preserve existing paid_at, set `processed_by` from input, update notes if provided

    **payment.queries.ts** — Implement:
    - `usePayments(params)` — query for list page
    - `usePaymentByBooking(bookingId)` — query for booking detail integration, enabled: !!bookingId
    - `useUpdatePaymentStatus()` — mutation, invalidates ['payments'] and ['payment-by-booking'] query keys

    **payment-form-schema.ts** — Define:
    - `mapPaymentError(error)` function following established pattern:
      - Auth expiry: 401, 403, PGRST301 → Vietnamese session expired message
      - 23505: check constraint name from **`error.message`** field (not `error.details` — PostgreSQL puts constraint name in message, details contains key/value pairs): `error.message?.includes('idx_payments_txn_ref_unique')` → "Mã giao dịch đã tồn tại cho phương thức này"; generic fallback → "Giá trị đã tồn tại" <!-- audit-added: 23505 discrimination uses message field per 07-02 audit finding -->
      - 23514 → "Dữ liệu không hợp liệu (vi phạm ràng buộc kiểm tra)" — covers paid_at/refunded_at CHECK constraints
      - default → "Thao tác thất bại. Vui lòng thử lại."

    **index.ts** — Export all types, API functions, query hooks, and constants (PAYMENT_STATUSES, PAYMENT_METHODS).

    **Important:** Follow the exact entity pattern from `src/entities/booking/` — same file structure, same export pattern, same query key convention.
