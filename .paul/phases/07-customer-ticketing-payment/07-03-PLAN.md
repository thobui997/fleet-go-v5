<objective>
## Goal
Deliver visual seat map rendering, ticket check-in operations with booking code lookup, and QR code generation/display for tickets — completing the operational ticket lifecycle.

## Purpose
These features close the gap between booking creation (07-02) and real-world ticket operations. The seat map gives dispatchers visual occupancy awareness. Check-in enables drivers/assistants to validate boarding. QR codes enable fast, error-free ticket verification and printing.

## Output
- SeatMap reusable component rendering vehicle seat layouts with booked/available states
- Ticket check-in page (/check-in) with booking code lookup and status transitions
- QR code generation on ticket creation and display in booking detail + print view
- Sidebar + router integration for check-in route
</objective>

<acceptance_criteria>

## AC-1: Seat Map Visualization
```gherkin
Given a booking with a trip that has a vehicle with a seat_layout (e.g., floor_1: { rows: 5, seats_per_row: 4 })
When the user opens the booking detail dialog
Then a visual seat map renders showing all seats in a grid layout per floor
And booked seats are highlighted in a distinct color from available seats
And a legend explains the color coding
```

## AC-2: Ticket Check-in by Booking Code
```gherkin
Given a booking with active tickets exists (booking_code "BKG-XXXXX")
When the operator navigates to /check-in and enters the booking code
Then the booking info (customer, trip route, departure time) is displayed
And each active ticket shows seat number, passenger name, and a "Check-in" button
When the operator clicks "Check-in" on a ticket
Then the ticket status transitions from 'active' to 'used'
And a success toast confirms the check-in
And the ticket row updates to show "Đã check-in" status
```

## AC-2b: Cancelled Booking Check-in Guard <!-- audit-added -->
```gherkin
Given a booking with status 'cancelled' or 'refunded'
When the operator enters the booking code on the check-in page
Then a warning banner shows "Đặt vé đã bị hủy" or "Đặt vé đã hoàn tiền"
And all check-in buttons are disabled
```

## AC-3: QR Code Generation and Display
```gherkin
Given a new booking is created with tickets
Then each ticket receives a qr_code value in format "{booking_code}-{seat_number}" (e.g., "BKG-ABCDE-A01")
When the user views booking detail
Then each ticket row shows a QR code icon button
When clicking the icon, a dialog shows the QR code image for that ticket
And the QR code dialog includes passenger name, seat number, and trip info for printing
```

## AC-4: Router and Sidebar Integration
```gherkin
Given the application is running
When navigating to /check-in
Then the CheckInPage renders (not PlaceholderPage)
And the sidebar shows "Check-in" in the Operations group with an appropriate icon
And the sidebar active state highlights correctly for /check-in
```

## AC-5: Error Handling
```gherkin
Given various error states
When check-in fails (ticket already used, booking cancelled, booking not found)
Then a Vietnamese error message is shown describing the specific issue
When a booking code search returns no results (PGRST116)
Then the error message is "Không tìm thấy đặt vé với mã này"
When a check-in operation fails because ticket is already processed (PGRST116 from mutation)
Then the error message is "Vé đã được check-in hoặc không còn hiệu lực" (distinct from search error)
When auth session expires during any operation
Then the 401/403/PGRST301 auth-expiry message is shown with retry guidance
```
