<objective>
## Goal
Redesign the login page with a modern split-screen layout featuring a branded cover area on the left and a focused form panel on the right.

## Purpose
Create a professional, branded first impression for users logging into FleetGo. The current centered card layout is functional but lacks visual impact. A split-screen design provides space for branding, improves visual hierarchy, and creates a more modern, enterprise-appropriate entry point.

## Output
- Redesigned login page with split-screen layout
- Responsive behavior (stacked on mobile, side-by-side on desktop)
- Cover area with branded image/illustration/gradient support
- Focused form panel with proper spacing and visual hierarchy
</objective>

<acceptance_criteria>

## AC-1: Split-screen Layout on Desktop
```gherkin
Given a user viewing the login page on desktop (≥1024px)
When the page loads
Then the display shows a two-column split-screen layout
  And the left section takes 50% width with full-height branded cover
  And the right section takes 50% width with the login form
  And both sections span the full viewport height
```

## AC-2: Responsive Behavior on Mobile
```gherkin
Given a user viewing the login page on mobile (<1024px)
When the page loads
Then the cover area is hidden
  And the form panel occupies 100% width
  And the layout centers vertically with appropriate padding
```

## AC-3: Cover Area Design
```gherkin
Given a user viewing the login page on desktop
When the cover area renders
Then it displays a branded background (gradient or image placeholder)
  And includes the FleetGo logo prominently
  And may include a welcome message or tagline
  And has proper visual hierarchy (logo > tagline)
```

## AC-4: Form Panel Structure
```gherkin
Given a user viewing the login form panel
When the panel renders
Then it shows the FleetGo title and description
  And displays email input with label
  And displays password input with label and visibility toggle
  And shows submit button with loading state
  And email input is auto-focused on page load
  And maintains existing validation and error handling
```

## AC-5: Loading and Error States
```gherkin
Given a user interacting with the login form
When authentication is in progress
Then the submit button shows "Đang đăng nhập..."
  And all form inputs are disabled
  And the existing skeleton loading state works during initial auth check
  And toast errors display for invalid credentials
```

## AC-6: Accessibility Preservation
```gherkin
Given a user using assistive technologies
When the login page renders
Then all form fields maintain proper labels
  And keyboard navigation works (Tab, Enter)
  And focus indicators use visible ring utilities (focus-visible:ring-2)
  And error messages are screen-reader accessible
  And email input receives focus on page load
```

## AC-7: Password Visibility Toggle
```gherkin
Given a user entering their password
When they click the eye icon in the password field
Then the password toggles between masked and visible states
  And the icon changes between Eye and EyeOff
  And focus remains in the password field
```
