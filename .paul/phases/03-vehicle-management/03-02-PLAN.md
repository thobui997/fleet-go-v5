<objective>
## Goal
Implement full CRUD for Vehicles — the second feature module. Covers list, create, edit, delete, FK dropdown to vehicle_types, and status management (active/maintenance/retired).

## Purpose
Vehicles are the core fleet entity. Every trip, maintenance log, and (eventually) booking chains back to a vehicle row. Shipping this unblocks 03-03 (maintenance logs) and every subsequent phase that needs a vehicle reference. This plan reuses the FSD entity+page pattern established by 03-01 with one new concern: foreign-key dropdowns backed by another entity's query hook.

## Output
- `src/entities/vehicle/` — entity slice with types and TanStack Query hooks
- `src/pages/vehicles/` — list page with status filter, form dialog (FK dropdown + status select), delete dialog
- Router wired to real page (replaces placeholder)
</objective>

## AC-1: Vehicles List Page
```gherkin
Given the user is authenticated and navigates to /vehicles
When the page loads
Then a paginated DataTable displays vehicles with columns: License Plate, Vehicle Type (resolved via FK join), Year, Status (colored Badge), Current Mileage, Next Maintenance Date, Created At, Actions
And pagination controls allow 10/20/50/100 per page
And the table supports client-side sorting on sortable columns
And a debounced search input filters by license_plate (case-insensitive ilike)
And a status filter (Select: All / Active / Maintenance / Retired) narrows the results server-side
```

## AC-2: Create Vehicle
```gherkin
Given the user clicks the "Add Vehicle" button on the list page
When a dialog opens and they fill in: vehicle_type_id (Select populated from useVehicleTypes, required),
  license_plate (required, trimmed), vin_number (optional),
  year_manufactured (optional, 1990..currentYear+1), status (Select, default 'active'),
  current_mileage (optional, >= 0), last_maintenance_date (optional, date),
  next_maintenance_date (optional, date), notes (optional)
And they submit the form
Then a new vehicles row is created in Supabase
And the list refreshes to show the new entry
And a success toast is displayed
```

## AC-3: Edit Vehicle
```gherkin
Given the user clicks the edit action on an existing vehicle row
When the edit dialog opens pre-populated with current values (dates serialized to YYYY-MM-DD strings; nullable fields to empty string)
And they modify fields and submit
Then the vehicles row is updated in Supabase
And the list refreshes to reflect changes
And a success toast is displayed
```

## AC-4: Delete Vehicle
```gherkin
Given the user clicks the delete action on a vehicle row
When a confirmation dialog appears showing the license plate
And the user confirms deletion
Then the vehicles row is deleted from Supabase
And the list refreshes
And a success toast is displayed
Or if a dependent row references it (FK RESTRICT from trips/maintenance_logs → 23503), a mapped error toast is displayed
```

## AC-5: Form Validation
```gherkin
Given the user submits the vehicle form with invalid data
When vehicle_type_id is empty, license_plate is empty, status is not one of the enum values,
  year_manufactured is outside [1990..currentYear+1], current_mileage is negative, or a date field is not a valid date
Then inline validation errors appear on the relevant fields with Vietnamese messages
And the form does not submit
```

## AC-6: Uniqueness Error Mapping (license_plate / vin_number)
```gherkin
Given the user creates or edits a vehicle with a license_plate or vin_number that already exists
When Supabase returns a unique constraint violation (23505)
Then a user-friendly Vietnamese error toast is displayed — license_plate clash → "Biển số xe đã tồn tại"; vin_number clash → "Số VIN đã tồn tại"
And the form remains open so the user can correct the value
```

## AC-7: Supabase Error Mapping (shared helper)
```gherkin
Given any Supabase mutation (create, update, delete) fails
When the error is a known PostgreSQL code (23505 unique violation, 23503 FK violation, 23514 check violation)
Then a mapped, user-friendly Vietnamese message is displayed — never a raw database error string
And for unknown errors, a generic "Thao tác thất bại. Vui lòng thử lại." is shown
```

## AC-8: Status Column Rendering
```gherkin
Given the vehicles list renders
When a row's status is read
Then the Status column renders a colored Badge via the VehicleStatusBadge component:
  active → green (success variant); maintenance → yellow/amber (warning variant); retired → gray (secondary variant)
And the Vietnamese label displayed is: "Đang hoạt động" / "Đang bảo trì" / "Đã ngừng sử dụng"
```

## AC-9: List Fetch Error State <!-- audit-added S7 -->
```gherkin
Given the user navigates to /vehicles
When useVehicles returns isError = true (Supabase outage, network failure, or auth expiry)
Then the page displays an inline error message — "Không thể tải danh sách xe. Vui lòng thử lại." — with a retry button
And the retry button calls refetch() and retries the query
And an empty table is NOT rendered when the query has failed
```

## AC-10: Dialog Close Guard During Pending Mutation <!-- audit-added S8 -->
```gherkin
Given a form dialog or delete dialog is open and a mutation is in flight (isPending = true)
When the user clicks the backdrop, presses Escape, or clicks Cancel
Then the dialog remains open until the mutation settles (success or error)
And no mid-flight cancellation occurs
```

    - `fetchVehicles(params: VehicleListParams)` — SELECT with `vehicle_type:vehicle_types ( id, name )` join; applies `.ilike('license_plate', \`%${search}%\`)` when search non-empty, `.eq('status', status)` when status provided; uses `.range()` for pagination and `{ count: 'exact' }`; orders by created_at desc; returns `{ data: VehicleWithType[], count: number }`
    - `fetchVehicle(id: string)` — SELECT single row by id (no join needed for form pre-fill; form stores vehicle_type_id)
    - `createVehicle(input: VehicleInsert)` — INSERT returning row
    - `updateVehicle(id: string, input: VehicleUpdate)` — UPDATE by id returning row
    - `deleteVehicle(id: string)` — DELETE by id
    - All functions throw the Supabase error object on failure so TanStack Query's onError receives it (same pattern as vehicle-type.api.ts)

    **1c. Create TanStack Query hooks** (`api/vehicle.queries.ts`):
    - `useVehicles(params)` — useQuery, queryKey: `['vehicles', params]`
    - `useVehicle(id)` — useQuery, queryKey: `['vehicles', id]`, `enabled: !!id`
    - `useCreateVehicle()` / `useUpdateVehicle()` / `useDeleteVehicle()` — useMutation, each invalidates `['vehicles']` on success via `queryClient.invalidateQueries({ queryKey: ['vehicles'] })`
    - Import from `@tanstack/react-query`

    **1d. Wire up public API** (`index.ts`):
    - Export types: Vehicle, VehicleWithType, VehicleInsert, VehicleUpdate, VehicleListParams, VehicleStatus, and the VEHICLE_STATUSES constant
    - Export hooks: useVehicles, useVehicle, useCreateVehicle, useUpdateVehicle, useDeleteVehicle
    - Update `src/entities/index.ts`: add `export * from './vehicle';`

    Avoid: Do NOT generate Supabase types via CLI — define manually. Do NOT introduce a global store; TanStack Query is the cache. Do NOT cross-import from @pages/vehicle-types; entity only depends on @entities/vehicle-type (per FSD).
