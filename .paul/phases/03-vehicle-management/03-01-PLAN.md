<objective>
## Goal
Implement full CRUD for Vehicle Types — the first feature module in the application. Establishes the FSD entity/page pattern that all subsequent phases will follow.

## Purpose
Vehicle Types are the foundational fleet entity. Every vehicle references a type (FK with RESTRICT). Building this first unblocks the Vehicles and Maintenance Logs plans. As the first feature module, the patterns established here (entity types, Supabase API layer, TanStack Query hooks, form validation, list page with DataTable) become the template for all future CRUD modules.

## Output
- `src/entities/vehicle-type/` — entity slice with types and API hooks
- `src/pages/vehicle-types/` — list page, form dialog, delete dialog
- Router wired to real page (replaces placeholder)
</objective>

<acceptance_criteria>

## AC-1: Vehicle Types List Page
```gherkin
Given the user is authenticated and navigates to /vehicle-types
When the page loads
Then a paginated DataTable displays vehicle types with columns: Name, Description, Total Seats, Total Floors, Amenities, Created At
And pagination controls allow 10/20/50/100 per page
And the table supports client-side sorting on sortable columns
```

## AC-2: Create Vehicle Type
```gherkin
Given the user clicks the "Add Vehicle Type" button on the list page
When a dialog opens and they fill in: name (required), description, total_floors (required, >=1), total_seats (required, >=1), seat_layout (required, valid JSON object), amenities (optional, comma-separated tags)
And they submit the form
Then a new vehicle_types row is created in Supabase
And the list refreshes to show the new entry
And a success toast is displayed
```

## AC-3: Edit Vehicle Type
```gherkin
Given the user clicks the edit action on an existing vehicle type row
When the edit dialog opens pre-populated with current values
And they modify fields and submit
Then the vehicle_types row is updated in Supabase
And the list refreshes to reflect changes
And a success toast is displayed
```

## AC-4: Delete Vehicle Type
```gherkin
Given the user clicks the delete action on a vehicle type row
When a confirmation dialog appears showing the type name
And the user confirms deletion
Then the vehicle_types row is deleted from Supabase (succeeds only if no vehicles reference it — FK RESTRICT)
And the list refreshes
And a success toast is displayed
Or if vehicles reference it, an error toast explains the constraint
```

## AC-5: Form Validation
```gherkin
Given the user submits the vehicle type form with invalid data
When name is empty, total_seats < 1, total_floors < 1, or seat_layout is not a valid JSON object
Then inline validation errors appear on the relevant fields
And the form does not submit
```

## AC-6: Duplicate Name Handling <!-- audit-added -->
```gherkin
Given the user creates or edits a vehicle type with a name that already exists
When Supabase returns a unique constraint violation (23505)
Then a user-friendly error toast is displayed (e.g., "A vehicle type with this name already exists")
And the form remains open so the user can correct the name
```

## AC-7: Supabase Error Mapping <!-- audit-added -->
```gherkin
Given any Supabase mutation (create, update, delete) fails
When the error is a known PostgreSQL code (23505 unique violation, 23503 FK violation)
Then a mapped, user-friendly message is displayed — never a raw database error string
And for unknown errors, a generic "Operation failed. Please try again." message is shown
```

    - `VehicleTypeInsert` type for create (omit id, created_at, updated_at)

    - `VehicleTypeListParams` for paginated queries: page, pageSize, search (optional)

    **1b. Create Supabase API functions** (`api/vehicle-type.api.ts`):
    - `fetchVehicleTypes(params: VehicleTypeListParams)` — paginated SELECT with optional ilike search on name, returns { data, count }
    - `fetchVehicleType(id: string)` — single row by id
    - `createVehicleType(input: VehicleTypeInsert)` — INSERT returning row
    - `updateVehicleType(id: string, input: VehicleTypeUpdate)` — UPDATE by id returning row
    - `deleteVehicleType(id: string)` — DELETE by id
    - Import supabase from `@shared/api`
    - Use `.range()` for pagination based on page/pageSize
    - Use `{ count: 'exact' }` on list query for total count
    - All functions must throw on error (check `error` from Supabase response and throw it) so TanStack Query's onError receives a typed error <!-- audit-added -->

    **1c. Create TanStack Query hooks** (`api/vehicle-type.queries.ts`):
    - `useVehicleTypes(params)` — useQuery wrapping fetchVehicleTypes, queryKey: ['vehicle-types', params]
    - `useVehicleType(id)` — useQuery wrapping fetchVehicleType, queryKey: ['vehicle-types', id], enabled: !!id
    - `useCreateVehicleType()` — useMutation wrapping createVehicleType, invalidates ['vehicle-types'] on success
    - `useUpdateVehicleType()` — useMutation wrapping updateVehicleType, invalidates ['vehicle-types'] on success
    - `useDeleteVehicleType()` — useMutation wrapping deleteVehicleType, invalidates ['vehicle-types'] on success
    - Import from `@tanstack/react-query`

    **1d. Wire up public API** (`index.ts`):
    - Export types: VehicleType, VehicleTypeInsert, VehicleTypeUpdate, VehicleTypeListParams
    - Export hooks: useVehicleTypes, useVehicleType, useCreateVehicleType, useUpdateVehicleType, useDeleteVehicleType
    - Update `src/entities/index.ts` to re-export from `./vehicle-type`

    Avoid: Do NOT generate Supabase types from the CLI — manually define types matching the schema. Do NOT add a global state store — TanStack Query is the caching layer.
