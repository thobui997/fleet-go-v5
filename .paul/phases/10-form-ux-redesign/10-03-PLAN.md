<objective>
## Goal
Move the trip form from a dialog into a dedicated full page at `/trips/new` and `/trips/:id/edit`, with field grouping via `FormSection`, a dirty-state blocker, a loading/error skeleton on edit, and integration with the existing staff assignment page.

## Purpose
The trip form has 7 fields across 3 concern groups (route/vehicle selection, times, status/options) — it scrolls in the dialog and has DateTime input UX issues. Moving it to a full page gives space for clear section grouping, allows the user to navigate directly to an edit URL, and integrates cleanly with the existing staff assignment sub-page.

## Output
- `src/pages/trips/ui/trip-form-page.tsx` — new full-page form (create + edit modes)
- `src/shared/config/routes.ts` — two new TRIPS route constants
- `src/app/lib/router.tsx` — two new routes registered
- `src/pages/trips/ui/trips-page.tsx` — list page updated: "Thêm" → navigate, "Sửa" → navigate, dialog removed
- `src/pages/trips/index.ts` — exports updated
- `src/pages/trips/ui/trip-form-dialog.tsx` — deleted (no longer used)
</objective>

    Logic:
    - Use `useParams<{ id: string }>()` from react-router-dom; if `id` is present → edit mode
    - In edit mode: call `useTrip(id)` from `@entities/trip`
    - Use `useEffect` to `reset(...)` the form when trip data arrives (same pattern as dialog)
    - Submit: call `createMutation.mutateAsync` or `updateMutation.mutateAsync`; on success → `toast` then **call `reset()` to clear `isDirty` before calling `navigate(ROUTES.TRIPS)`** — this prevents `useBlocker` from intercepting the post-submit redirect; on error → `toast` with `mapTripError` (which handles auth-expiry 401/403/PGRST301)

    - Reuse `tripFormSchema`, `serializeToInsert`, `mapTripError`, `FK_DROPDOWN_PAGE_SIZE` from `../model/trip-form-schema`
    - `useBlocker`: import from `react-router-dom`; use the callback form with a pathname guard to prevent same-page navigation from triggering the blocker. **Blocker condition: `isDirty && !isPending && currentLocation.pathname !== nextLocation.pathname`** — `!isPending` ensures blocker fires during submission to prevent duplicate actions:
      ```ts
      const blocker = useBlocker(
        ({ currentLocation, nextLocation }) =>
          isDirty && !isPending && currentLocation.pathname !== nextLocation.pathname
      );
      ```
      Render a `<Dialog>` when `blocker.state === 'blocked'` with "Bạn có dữ liệu chưa lưu. Thoát không?" and two buttons: "Ở lại" (`blocker.reset()`) + "Thoát" (`blocker.proceed()`)
      <!-- audit-added: explicit !isPending in blocker condition to prevent navigation during submission -->
    - FK dropdown truncation warnings: same pattern as dialog for both routes and vehicles
    - FK dropdown empty state handling: When routes.length === 0, show "Chưa có tuyến đường nào — tạo tuyến đường trước ở /routes" and disable form submission. Same pattern for vehicles.
    <!-- audit-added: empty state handling prevents user being stuck without route/vehicle options -->
    - Datetime validation for new trips: departure_time must be in the future (arrival must also be after departure, already enforced by schema superRefine)
    <!-- audit-added: business rule enforcement for new trip creation -->
    - Context-aware fetch error message — define inline in the component:
      ```ts
      function mapFetchError(error: unknown): string {
        const e = error as { code?: string; status?: number } | null;
        if (e?.code === 'PGRST116' || e?.status === 406)
          return 'Không tìm thấy chuyến đi.';
        if (e?.status === 401 || e?.status === 403 || e?.code === 'PGRST301')
          return 'Phiên đăng nhập đã hết hạn. Vui lòng đăng nhập lại.';
        return 'Không thể tải chuyến đi. Vui lòng thử lại.';
      }
      ```

    - Use `hasInitializedRef` pattern to guard background refetch overwrites: set to true after first reset in edit mode; skip subsequent resets
