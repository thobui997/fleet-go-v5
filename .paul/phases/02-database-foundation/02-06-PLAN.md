<objective>
## Goal
Create Row-Level Security (RLS) policies for all 16 public tables using dynamic permission checks via `user_roles` → `roles.permissions` JSONB lookups, enforcing role-based access at the database level.

## Purpose
RLS is the project's sole authorization mechanism (per PROJECT.md: "Supabase RLS policies for data access control"). Without RLS, every table is open to any authenticated user via PostgREST. This plan locks down all data access to match the dynamic role system designed in 02-01.

## Output
- `supabase/migrations/20260414110000_rls_helpers.sql` — SECURITY DEFINER helper functions
- `supabase/migrations/20260414110001_rls_policies.sql` — ALTER TABLE ENABLE RLS + CREATE POLICY for all 16 tables
</objective>

## AC-1: RLS Helper Functions
```gherkin
Given a user authenticated via Supabase Auth
When the database evaluates has_permission('vehicles:read')
Then it returns true if the user has a role whose permissions JSONB contains '"vehicles:read"' or '"*"', false otherwise
```

## AC-2: RLS Enabled on All Tables
```gherkin
Given all 16 public tables (profiles, roles, user_roles, employees, vehicle_types, vehicles, maintenance_logs, stations, routes, route_stops, trips, trip_staff, customers, bookings, tickets, payments)
When RLS is enabled
Then every table has force_row_security = true and at least one policy defined
```

## AC-3: Permission-Gated Access
```gherkin
Given a user with role 'fleet_manager' (permissions: vehicles:read, vehicles:write, vehicle_types:read, vehicle_types:write, maintenance:read, maintenance:write)
When they query the bookings table
Then the query returns zero rows (no bookings:read permission)
And when they query the vehicles table
Then the query returns all vehicles (has vehicles:read permission)
```

## AC-4: Self-Access Patterns
```gherkin
Given any authenticated user
When they query profiles where id = auth.uid()
Then they see their own profile row
And when they query employees where user_id = auth.uid()
Then they see their own employee record (if one exists)
And when a user has no explicit permission for a reference table (roles, stations, routes, route_stops, vehicle_types)
Then they can still SELECT from it (reference data is readable by all authenticated users)
```

## AC-5: Audit-Attribution Enforcement <!-- audit-added -->
```gherkin
Given a user with bookings:write permission
When they INSERT a booking with created_by set to another user's UUID
Then the INSERT is rejected by the WITH CHECK policy
And when they INSERT a booking with created_by = auth.uid() or created_by IS NULL
Then the INSERT succeeds
And the same enforcement applies to tickets.issued_by and payments.processed_by on INSERT
```

## AC-6: Helper Function Access Control <!-- audit-added -->
```gherkin
Given an unauthenticated (anon) connection
When it calls has_permission() or is_admin()
Then the call is denied (EXECUTE revoked from PUBLIC, granted only to authenticated)
```

<task type="auto">
  <name>Task 1: Create RLS helper functions</name>
  <files>supabase/migrations/20260414110000_rls_helpers.sql</files>
  <action>
    Create a migration file with two SECURITY DEFINER helper functions that RLS policies will reference.
    Both functions must use `SET search_path = public` to prevent search path injection (same pattern as handle_new_user).

    **Function 1: `public.has_permission(required_permission text) RETURNS boolean`**
    - Uses `auth.uid()` to identify current user
    - Queries `user_roles` JOIN `roles` to check if ANY of the user's roles have `permissions @> jsonb_build_array(required_permission)` OR `permissions @> '["*"]'::jsonb`
    - Returns `false` if `auth.uid()` is NULL (anonymous)
    - SECURITY DEFINER so it can read user_roles and roles regardless of their own RLS
    - STABLE (no side effects, same result within transaction)

    **Function 2: `public.is_admin() RETURNS boolean`**
    - Shorthand: calls `has_permission('*')` — but more efficient: directly checks `permissions @> '["*"]'::jsonb`
    - Uses `auth.uid()`, returns false if NULL
    - SECURITY DEFINER, STABLE, SET search_path = public

    Add COMMENT ON FUNCTION for both.
    Add a migration header comment block matching existing migration style.

    <!-- audit-added: REVOKE/GRANT and performance note -->
    **Access control on helper functions:**
    After creating both functions, REVOKE EXECUTE from PUBLIC and GRANT EXECUTE to authenticated.
    This prevents the anon role from calling permission-check functions.
    ```sql
    REVOKE EXECUTE ON FUNCTION public.has_permission(text) FROM PUBLIC;
    GRANT EXECUTE ON FUNCTION public.has_permission(text) TO authenticated;
    REVOKE EXECUTE ON FUNCTION public.is_admin() FROM PUBLIC;
    GRANT EXECUTE ON FUNCTION public.is_admin() TO authenticated;
    ```

    **Performance note (add as COMMENT ON FUNCTION):**
    Both functions are marked STABLE and do not reference row columns — PostgreSQL's planner
    can evaluate them once per query rather than per-row when used in RLS policy expressions.
  </action>
  <verify>

    - Both use SET search_path = public
    - has_permission accepts text, returns boolean
    - is_admin accepts no args, returns boolean
    - Both marked STABLE
    - REVOKE EXECUTE FROM PUBLIC present for both functions <!-- audit-added -->
    - GRANT EXECUTE TO authenticated present for both functions <!-- audit-added -->
  </verify>

<task type="auto">
  <name>Task 2: Create RLS policies for all 16 tables</name>
  <files>supabase/migrations/20260414110001_rls_policies.sql</files>
  <action>
    Create a migration file that enables RLS on all 16 public tables and creates named policies.
    Use `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and `ALTER TABLE ... FORCE ROW LEVEL SECURITY` for each table.

    **Policy naming convention:** `{table}_{operation}_{scope}` (e.g., `profiles_select_own`, `vehicles_select_permission`)

    **Two access tiers:**

    **Tier 1 — Reference tables (SELECT open to all authenticated, write restricted):**
    These tables contain reference/configuration data needed by UI joins. No sensitive data.

    | Table | SELECT | INSERT | UPDATE | DELETE |
    |-------|--------|--------|--------|--------|
    | profiles | auth.uid() IS NOT NULL | none (handle_new_user trigger is DEFINER) | auth.uid() = id (own row) OR is_admin() | none (cascade from auth.users) |
    | roles | auth.uid() IS NOT NULL | is_admin() | is_admin() | is_admin() |
    | user_roles | auth.uid() IS NOT NULL (needed for role display) | is_admin() | is_admin() | is_admin() |
    | vehicle_types | auth.uid() IS NOT NULL | has_permission('vehicle_types:write') | has_permission('vehicle_types:write') | is_admin() |
    | stations | auth.uid() IS NOT NULL | has_permission('stations:write') | has_permission('stations:write') | is_admin() |
    | routes | auth.uid() IS NOT NULL | has_permission('routes:write') | has_permission('routes:write') | is_admin() |
    | route_stops | auth.uid() IS NOT NULL | has_permission('routes:write') | has_permission('routes:write') | is_admin() |

    **Tier 2 — Domain tables (read AND write permission-gated):**

    | Table | SELECT | INSERT | UPDATE | DELETE |
    |-------|--------|--------|--------|--------|
    | employees | has_permission('employees:read') OR (user_id = auth.uid()) | has_permission('employees:write') | has_permission('employees:write') OR (user_id = auth.uid() AND only non-sensitive cols) | is_admin() |
    | vehicles | has_permission('vehicles:read') | has_permission('vehicles:write') | has_permission('vehicles:write') | is_admin() |
    | maintenance_logs | has_permission('maintenance:read') | has_permission('maintenance:write') | has_permission('maintenance:write') | is_admin() |
    | trips | has_permission('trips:read') | has_permission('trips:write') | has_permission('trips:write') | is_admin() |
    | trip_staff | has_permission('trips:read') OR (employee_id IN (SELECT id FROM employees WHERE user_id = auth.uid())) | has_permission('trips:write') | has_permission('trips:write') | is_admin() |
    | customers | has_permission('customers:read') | has_permission('customers:write') | has_permission('customers:write') | is_admin() |
    | bookings | has_permission('bookings:read') | has_permission('bookings:write') | has_permission('bookings:write') | is_admin() |
    | tickets | has_permission('bookings:read') | has_permission('bookings:write') | has_permission('bookings:write') | is_admin() |
    | payments | has_permission('payments:read') | has_permission('payments:write') | has_permission('payments:write') | is_admin() |

    **Important implementation notes:**
    - For `employees` self-access UPDATE: use WITH CHECK to prevent users from changing their own `user_id`, `is_active`, or `hire_date` — only allow updating contact-related fields. Simplification: skip self-update for now; only permission-holders can update employees. Self-access is SELECT only.
    - For `trip_staff` self-access: the subquery `SELECT id FROM employees WHERE user_id = auth.uid()` finds the current user's employee_id to match against trip_staff.employee_id. This lets drivers/assistants see their own assignments even without explicit trips:read (they do have it in seed, but this is defense-in-depth).
    - The `is_admin()` check on DELETE for all tables ensures only administrators can delete records. This is a safety net — most DELETE operations should be soft-deletes via status changes.
    - All policies use `TO authenticated` role (not public/anon).
    - Add COMMENT ON POLICY is not supported in PostgreSQL — use inline SQL comments instead.

    <!-- audit-added: WITH CHECK for audit-attribution columns -->
    **Audit-attribution WITH CHECK constraints (MUST-HAVE):**
    INSERT policies for tables with audit-trail attribution columns MUST include WITH CHECK
    to prevent staff from attributing record creation to another user:
    - `bookings` INSERT: `WITH CHECK (has_permission('bookings:write') AND (created_by IS NULL OR created_by = auth.uid()))`
    - `tickets` INSERT: `WITH CHECK (has_permission('bookings:write') AND (issued_by IS NULL OR issued_by = auth.uid()))`
    - `payments` INSERT: `WITH CHECK (has_permission('payments:write') AND (processed_by IS NULL OR processed_by = auth.uid()))`
    Note: cancelled_by enforcement on UPDATE is deferred to 02-07 immutability triggers (cannot cleanly enforce column-level change detection via RLS WITH CHECK alone).

    <!-- audit-added: schema-qualify all function calls -->
    **Schema-qualify all helper function calls in policy expressions:**
    Use `public.has_permission()` and `public.is_admin()` (not unqualified names) in all
    CREATE POLICY statements. Prevents search_path manipulation from resolving to a
    different function.

    Group policies by table in the migration file with clear section headers matching the existing migration style.
  </action>
