<objective>
## Goal
Create the booking schema with `customers`, `bookings`, `tickets`, and `payments` tables — the revenue-generating core of the system. Includes auto-generated booking codes (BKG-NNNNN via sequence), seat double-booking prevention via partial unique index, 1:1 booking/payment relationship, realistic Vietnamese seed data.

## Purpose
Bookings are where revenue happens: customers buy tickets for specific seats on trips, and payments close the loop. This schema is the last domain schema in Phase 2 — after this, 02-06 (RLS) and 02-07 (triggers/functions) can apply security and business logic across all tables. It enables Phase 7 (Customer, Ticketing & Payment) to build booking workflows, seat selection UI, and payment tracking on a solid data foundation.

## Output
- `supabase/migrations/20260414100000_booking_schema.sql` — customers, bookings, tickets, payments tables with FKs, constraints, indexes, comments, booking_code sequence
- `supabase/migrations/20260414100001_booking_triggers.sql` — updated_at triggers for all four tables
- `supabase/seed.sql` — appended booking seed data (4 customers, 3 bookings, 5 tickets, 3 payments)
</objective>

  <action>
    Create migration file following established patterns (see trip_schema.sql and fleet_schema.sql).

    **Sequence for booking codes (before tables):**
    ```sql
    create sequence public.booking_code_seq start 1;
    comment on sequence public.booking_code_seq is 'Monotonic sequence for generating human-readable booking codes (BKG-NNNNN).';
    ```

    **customers table:**
    - id: uuid PK default gen_random_uuid()
    - full_name: text NOT NULL CHECK (length(trim(full_name)) > 0)
    - phone_number: text NOT NULL UNIQUE
    - email: text UNIQUE (nullable)
    - date_of_birth: date (nullable)
    - gender: text CHECK (gender IS NULL OR gender IN ('male','female','other'))
    - id_card_number: text UNIQUE (nullable — Vietnamese CMND/CCCD)
    - address: text
    - loyalty_points: int NOT NULL DEFAULT 0 CHECK (loyalty_points >= 0)
    - notes: text
    - created_at, updated_at: timestamptz NOT NULL DEFAULT now()

    **Indexes on customers:**
    - idx_customers_phone_number on phone_number (covered by UNIQUE but explicit for query planner clarity — actually unique constraints create indexes automatically, so skip redundant one)
    - idx_customers_email on email (nullable; useful for null-friendly search — actually UNIQUE creates index, skip)
    - Note: UNIQUE creates supporting indexes automatically. Only add explicit indexes for non-unique query patterns.
    - idx_customers_full_name_trgm: deferred to later phase if needed (requires pg_trgm extension)

    **bookings table:**
    - id: uuid PK default gen_random_uuid()
    - booking_code: text NOT NULL UNIQUE DEFAULT ('BKG-' || lpad(nextval('public.booking_code_seq')::text, 5, '0'))
    - customer_id: uuid NOT NULL, FK → customers(id) ON DELETE RESTRICT
    - trip_id: uuid NOT NULL, FK → trips(id) ON DELETE RESTRICT
    - booking_date: timestamptz NOT NULL DEFAULT now()
    - status: text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','confirmed','cancelled','completed','refunded'))
    - total_amount: numeric(12,2) NOT NULL CHECK (total_amount >= 0)
    - passenger_count: int NOT NULL CHECK (passenger_count > 0)
    - created_by: uuid, FK → profiles(id) ON DELETE SET NULL (nullable — staff who made booking; SET NULL preserves booking history if staff profile deleted)
    - cancelled_at: timestamptz (nullable — set when booking is cancelled) <!-- audit-added -->
    - cancelled_by: uuid, FK → profiles(id) ON DELETE SET NULL (nullable — staff who cancelled booking) <!-- audit-added -->
    - notes: text
    - created_at, updated_at: timestamptz NOT NULL DEFAULT now()
    - CHECK constraint: cancelled_at IS NULL OR status = 'cancelled' (enforces timestamp is set only when booking is cancelled) <!-- audit-added -->
    - UNIQUE constraint on (id, trip_id) — supports composite FK from tickets, preventing ticket.trip_id drift from booking.trip_id <!-- audit-added: must-have -->

    **Indexes on bookings:**
    - idx_bookings_customer_id on customer_id
    - idx_bookings_trip_id on trip_id
    - idx_bookings_status on status
    - idx_bookings_booking_date on booking_date
    - idx_bookings_created_by on created_by
    - idx_bookings_cancelled_by on cancelled_by <!-- audit-added -->
    - Note: UNIQUE on (id, trip_id) creates supporting index automatically

    **tickets table:**
    - id: uuid PK default gen_random_uuid()
    - booking_id: uuid NOT NULL — part of composite FK (see below)
    - trip_id: uuid NOT NULL — part of composite FK (see below)
    - seat_number: text NOT NULL CHECK (length(trim(seat_number)) > 0)
    - passenger_name: text NOT NULL CHECK (length(trim(passenger_name)) > 0)
    - passenger_id_card: text
    - passenger_phone: text
    - price: numeric(12,2) NOT NULL CHECK (price >= 0)
    - status: text NOT NULL DEFAULT 'active' CHECK (status IN ('active','used','cancelled','refunded'))
    - qr_code: text (nullable — optional per PROJECT.md)
    - issued_by: uuid, FK → profiles(id) ON DELETE SET NULL (nullable — staff who issued ticket; audit trail) <!-- audit-added -->
    - created_at, updated_at: timestamptz NOT NULL DEFAULT now()

    **Foreign keys on tickets:**
    - booking_id FK → bookings(id) ON DELETE CASCADE (simple single-column FK)
    - trip_id FK → trips(id) ON DELETE RESTRICT (standalone FK for fast trip-centric queries)
    - **Composite FK (MUST-HAVE data integrity):** FOREIGN KEY (booking_id, trip_id) REFERENCES bookings(id, trip_id) ON DELETE CASCADE <!-- audit-added: must-have -->
      This composite FK enforces that a ticket's trip_id ALWAYS equals the referenced booking's trip_id. Without this, tickets.trip_id could drift, silently bypassing the seat double-booking partial unique index. Requires the UNIQUE(id, trip_id) constraint on bookings (added above).

    **Indexes on tickets:**
    - idx_tickets_booking_id on booking_id
    - idx_tickets_trip_id on trip_id
    - idx_tickets_status on status
    - idx_tickets_issued_by on issued_by <!-- audit-added -->
    - Partial unique index: CREATE UNIQUE INDEX idx_tickets_no_double_booking ON tickets (trip_id, seat_number) WHERE status IN ('active','used')
      Prevents selling the same seat twice on the same trip. Cancelled/refunded tickets don't block the seat.
    - COMMENT ON INDEX idx_tickets_no_double_booking IS 'Business rule: a seat on a trip can only be held by one active/used ticket. Cancelled or refunded tickets release the seat.'
    - Partial unique index: CREATE UNIQUE INDEX idx_tickets_qr_code_unique ON tickets (qr_code) WHERE qr_code IS NOT NULL <!-- audit-added -->
      Prevents duplicate QR codes across all tickets — boarding validation must be deterministic. NULLs are allowed (QR codes are optional per PROJECT.md).
    - COMMENT ON INDEX idx_tickets_qr_code_unique IS 'Business rule: QR codes must be unique across all tickets for deterministic boarding validation.' <!-- audit-added -->

    **payments table:**
    - id: uuid PK default gen_random_uuid()
    - booking_id: uuid NOT NULL UNIQUE, FK → bookings(id) ON DELETE RESTRICT (UNIQUE enforces 1:1 booking/payment relationship per PROJECT.md data model)
    - amount: numeric(12,2) NOT NULL CHECK (amount >= 0)
    - method: text NOT NULL CHECK (method IN ('cash','e_wallet','bank_transfer'))
    - status: text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','completed','failed','refunded'))
    - transaction_reference: text (nullable — e-wallet txn ID, bank ref, etc.)
    - paid_at: timestamptz (nullable — set when status → completed)
    - refunded_at: timestamptz (nullable — set when payment refunded; separate from paid_at to preserve original payment timeline) <!-- audit-added -->
    - processed_by: uuid, FK → profiles(id) ON DELETE SET NULL (nullable — staff who processed payment; audit trail for cash handling) <!-- audit-added -->
    - notes: text
    - created_at, updated_at: timestamptz NOT NULL DEFAULT now()
    - CHECK constraint: paid_at IS NULL OR status IN ('completed','refunded')  -- paid_at only meaningful when payment completed/refunded
    - CHECK constraint: refunded_at IS NULL OR status = 'refunded'  -- refunded_at requires refunded status <!-- audit-added -->

    **Indexes on payments:**
    - idx_payments_status on status
    - idx_payments_paid_at on paid_at
    - idx_payments_processed_by on processed_by <!-- audit-added -->
    - Partial unique index: CREATE UNIQUE INDEX idx_payments_txn_ref_unique ON payments (method, transaction_reference) WHERE transaction_reference IS NOT NULL <!-- audit-added -->
      Prevents duplicate transaction references per payment method — protects against webhook replay, double-callback, or manual re-entry of the same external transaction.
    - COMMENT ON INDEX idx_payments_txn_ref_unique IS 'Business rule: each (method, transaction_reference) pair must be unique to prevent duplicate logging of external payment events (webhook replay protection).' <!-- audit-added -->
    - Note: booking_id UNIQUE creates index automatically
