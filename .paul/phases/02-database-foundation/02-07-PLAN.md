<objective>
## Goal
Add database-level integrity triggers enforcing (a) set-once immutability of audit-trail columns across `bookings`, `tickets`, and `payments`, and (b) valid state transitions on `bookings.status`, closing enforcement gaps that RLS `WITH CHECK` cannot cover.

## Purpose
02-05 and 02-06 both deferred audit-column immutability to 02-07 because RLS `WITH CHECK` on `UPDATE` evaluates only the new row — it cannot compare OLD vs NEW to detect column-level change. RLS controls WHO can write; these triggers control WHAT can change once set. Without them, any user with write permission could silently rewrite `cancelled_by`, `issued_by`, `processed_by`, or flip a cancelled booking back to confirmed — corrupting the audit trail and the business-logic state machine.

## Output
- `supabase/migrations/20260414120000_integrity_triggers.sql` — 4 trigger functions + 4 triggers (1 on tickets, 1 on payments, 2 on bookings)
</objective>

<acceptance_criteria>

## AC-1: Bookings audit-column immutability
```gherkin
Given a booking row where any of {booking_code, created_by, cancelled_at, cancelled_by} is NOT NULL  <!-- audit-added: booking_code -->
When any role (including admin) executes UPDATE bookings SET <that column> = <different value or NULL>
Then the UPDATE is rejected with an exception identifying the immutable column
And the exception carries SQLSTATE 'FG001', DETAIL containing the booking id, and HINT containing auth.uid()  <!-- audit-added -->
And when the column is NULL and UPDATE sets it to a value for the first time
Then the UPDATE succeeds (set-once allows the initial assignment only)
```

## AC-2: Tickets & payments audit-column immutability
```gherkin
Given a ticket with issued_by set, or a payment with processed_by / paid_at / refunded_at set
When any role attempts to change those columns to a different value or NULL
Then the UPDATE is rejected with an exception identifying the immutable column
And the exception carries SQLSTATE 'FG002' (tickets) or 'FG003' (payments), DETAIL containing the row id, and HINT containing auth.uid()  <!-- audit-added -->
And first-time assignment from NULL to a value remains allowed
```

## AC-3: Booking status state-machine enforcement
```gherkin
Given a booking with status S_old and a proposed status S_new where S_old != S_new
When the UPDATE is executed
Then the transition is allowed only if (S_old, S_new) is one of:
  (pending, confirmed), (pending, cancelled),
  (confirmed, completed), (confirmed, cancelled),
  (completed, refunded)
And any other transition — including any change from 'cancelled' or 'refunded' (terminal states) — raises an exception
And the exception carries SQLSTATE 'FG004', DETAIL containing the booking id, and HINT containing auth.uid()  <!-- audit-added -->
```

## AC-4: Legal updates remain unobstructed
```gherkin
Given a booking in status 'pending' with cancelled_at / cancelled_by still NULL
When it is updated to status 'confirmed' (no-op on audit columns)
Then the UPDATE succeeds, and set_bookings_updated_at refreshes updated_at as before
And when a booking moves from 'confirmed' → 'cancelled' with cancelled_at = now() and cancelled_by = auth.uid() in the same UPDATE
Then the UPDATE succeeds (first-time set is permitted and state transition is legal)
```

<!-- audit-added: AC-5 -->
## AC-5: Observable, machine-classifiable exceptions
```gherkin
Given any integrity trigger raises an exception (AC-1, AC-2, or AC-3 violation)
When a client catches the error
Then SQLSTATE is one of the distinct codes: 'FG001' (bookings audit), 'FG002' (tickets audit), 'FG003' (payments audit), 'FG004' (booking status transition)
And the error DETAIL includes the affected row's primary key ("id=<uuid>")
And the error HINT includes the attempted actor ("attempted_by=<auth.uid() or NULL>")
And the MESSAGE remains human-readable (for logs) but is NOT required for client-side classification
```

         using errcode = 'FG001',`  <!-- audit-added: distinct SQLSTATE per violation class -->
      `         detail  = format('id=%s', NEW.id),`  <!-- audit-added: row PK for post-incident trace -->
      `         hint    = format('attempted_by=%s', coalesce(auth.uid()::text, 'NULL'));`  <!-- audit-added: actor identity for attribution -->
    - IS DISTINCT FROM handles the NULL vs value case correctly and rejects NULL-ing a non-NULL column
    - Return NEW at the end

    Add COMMENT ON FUNCTION explaining: set-once enforcement for audit-trail columns including booking_code; pairs with RLS WITH CHECK on INSERT from 02-06; raises SQLSTATE FG001.

    Drop-and-create trigger for idempotency <!-- audit-added -->:
    ```sql
    drop trigger if exists guard_bookings_audit_immutable on public.bookings;
    create trigger guard_bookings_audit_immutable
      before update on public.bookings
      for each row
      when (OLD.booking_code is not null OR OLD.created_by is not null OR OLD.cancelled_at is not null OR OLD.cancelled_by is not null)
      execute function public.guard_bookings_audit_immutable();
    ```
    - Note: `OLD.booking_code IS NOT NULL` is always true (NOT NULL column with DEFAULT); it's kept in the WHEN clause for documentation symmetry.
    - The rest of the WHEN clause short-circuits cheaply when other audit columns are still NULL (common case for pending bookings).

    Avoid:
    - Don't use SECURITY DEFINER (no privileged reads needed).
    - Don't touch `updated_at` — existing `set_bookings_updated_at` handles that.
    - Don't mix status-transition logic into this function — kept in Task 3 for clarity.

## OPERATIONAL NOTES (audit-added)
<!-- audit-added: honest documentation of escape-hatch cost -->
- **Administrative corrections** to set-once audit columns (e.g., a legal correction to re-attribute `cancelled_by`) are NOT supported via standard SQL UPDATE by design. Such corrections require a DBA action: temporarily drop or disable the relevant trigger, make the correction with a recorded justification, and re-create the trigger. The rollback script at the end of the migration file documents the DROP statements needed.
- **Cross-table invariants** (e.g., `bookings.status='refunded'` should imply `payments.status='refunded'`) are explicitly OUT OF SCOPE here and tracked as a known gap to be closed when the payments FSM is designed in Phase 7.
