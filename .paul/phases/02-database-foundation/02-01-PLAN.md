<objective>
## Goal
Create the core database schema (profiles, roles, user_roles, employees) with proper constraints, indexes, and essential triggers — the foundation all future feature schemas depend on.

## Purpose
Phase 1 established authentication via Supabase GoTrue, but the database has no application tables yet. Every feature module (vehicles, routes, employees, trips, bookings) references users and roles. This plan creates those foundational tables with proper foreign keys, indexes, and automated behaviors (auto-profile creation, timestamp updates) before any feature work begins.

## Output
- SQL migration files for core schema in `supabase/migrations/`
- `supabase/seed.sql` with realistic Vietnamese-context seed data (separate from migrations)
- `profiles` table extending `auth.users` — single source of truth for user data
- `roles` table for dynamic role management with GIN-indexed JSONB permissions
- `user_roles` junction table with composite primary key
- `employees` table for employment-specific data only (no user data duplication)
- Essential triggers: auto-profile on signup (with secure search_path), auto-update `updated_at`
</objective>

## AC-1: Core Tables Created with Proper Schema
```gherkin
Given the migration SQL is applied to the Supabase database
When the database is queried
Then the following tables exist with correct columns and constraints:
  - profiles (id FK→auth.users PK, email, full_name, phone, avatar_url, created_at, updated_at)
  - roles (id UUID PK, name UNIQUE, description, permissions JSONB, created_at, updated_at)
  - user_roles (user_id FK→profiles, role_id FK→roles, created_at, PRIMARY KEY(user_id, role_id)) — no surrogate id
  - employees (id UUID PK, user_id FK→profiles NULLABLE UNIQUE ON DELETE SET NULL, hire_date, license_number UNIQUE, license_expiry, is_active, created_at, updated_at) — no full_name, email, or phone (those come from profiles)
And roles.permissions has a CHECK constraint ensuring JSONB array type
And profiles is the single source of truth for user contact data (full_name, email, phone)
And employees stores only employment-specific data (hire dates, license, active status)
And user_roles uses a composite primary key instead of a surrogate id column
And all timestamp columns use timestamptz with default now()
```

## AC-2: Indexes Support Common Query Patterns
```gherkin
Given the core tables exist
When analyzing query performance
Then the following indexes exist:
  - profiles: idx on email (for login lookups)
  - roles: GIN idx on permissions (for JSONB containment queries, e.g. permissions @> '["vehicles:read"]')
  - user_roles: composite PK on (user_id, role_id) — covers join queries
  - user_roles: idx on role_id (for "all users with role X" queries)
  - employees: idx on user_id (for profile→employee join)
  - employees: idx on is_active (for filtering active/inactive)
  - employees: idx on license_expiry (for expiry alert queries)
```

       ```sql
       create table public.profiles (
         id uuid references auth.users on delete cascade primary key,
         email text not null,
         full_name text,
         phone text,
         avatar_url text,
         created_at timestamptz not null default now(),
         updated_at timestamptz not null default now()
       );

       create index idx_profiles_email on public.profiles(email);

       comment on table public.profiles is 'Application user profiles — extends Supabase auth.users. Single source of truth for user contact data.';
       comment on column public.profiles.id is 'References auth.users.id — set by trigger on user signup';
       ```

       **roles** — Dynamic roles (not hardcoded):
       ```sql
       create table public.roles (
         id uuid not null default gen_random_uuid() primary key,
         name text not null unique,
         description text,
         permissions jsonb not null default '[]'::jsonb
           check (jsonb_typeof(permissions) = 'array'),
         created_at timestamptz not null default now(),
         updated_at timestamptz not null default now()
       );

       create index idx_roles_permissions on public.roles using gin (permissions);

       comment on table public.roles is 'Dynamic role definitions — permissions stored as JSONB array of strings';
       comment on column public.roles.permissions is 'JSONB array of permission strings, e.g. ["vehicles:read", "vehicles:write"]. Queried with @> operator.';
       ```

       **user_roles** — Junction table with composite primary key (no surrogate id):
       ```sql
       create table public.user_roles (
         user_id uuid not null references public.profiles(id) on delete cascade,
         role_id uuid not null references public.roles(id) on delete cascade,
         created_at timestamptz not null default now(),
         primary key (user_id, role_id)
       );

       create index idx_user_roles_role_id on public.user_roles(role_id);

       comment on table public.user_roles is 'Junction table — links users to their assigned roles. Composite PK eliminates redundant surrogate key.';
       ```

       **employees** — Employment-specific data only (no user data duplication):
       ```sql
       create table public.employees (
         id uuid not null default gen_random_uuid() primary key,
         user_id uuid unique references public.profiles(id) on delete set null,
         hire_date date,
         license_number text unique,
         license_expiry date,
         is_active boolean not null default true,
         created_at timestamptz not null default now(),
         updated_at timestamptz not null default now()
       );

       create index idx_employees_user_id on public.employees(user_id);
       create index idx_employees_is_active on public.employees(is_active);
       create index idx_employees_license_expiry on public.employees(license_expiry);

       comment on table public.employees is 'Staff employment records. User data (name, email, phone) comes from profiles via user_id FK. Join employees→profiles to get staff details.';
       comment on column public.employees.user_id is 'UNIQUE — enforces 1:1 relationship with profiles. Nullable for edge cases (PostgreSQL UNIQUE allows multiple NULLs).';
       comment on column public.employees.license_number is 'UNIQUE — two employees cannot share the same license. Nullable for non-driving staff.';
       comment on column public.employees.license_expiry is 'Used for 30-day expiry warning alerts in Phase 5';
       ```

  <name>Task 2: Create essential database triggers</name>
  <files>
    supabase/migrations/20260410120001_core_triggers.sql

  <action>
    1. Create `supabase/migrations/20260410120001_core_triggers.sql` with the following triggers:

       **Reusable updated_at function** — called by all tables:
       ```sql
       create or replace function public.handle_updated_at()
       returns trigger
       language plpgsql
       as $$
       begin
         new.updated_at = now();
         return new;
       end;
       $$;
       ```

       **updated_at triggers for all core tables:**
       ```sql
       create trigger set_profiles_updated_at
         before update on public.profiles
         for each row execute function public.handle_updated_at();

       create trigger set_roles_updated_at
         before update on public.roles
         for each row execute function public.handle_updated_at();

       create trigger set_user_roles_updated_at
         before update on public.user_roles
         for each row execute function public.handle_updated_at();

       create trigger set_employees_updated_at
         before update on public.employees
         for each row execute function public.handle_updated_at();
       ```

       **Auto-create profile on user signup (with secure search_path):**
       ```sql
       create or replace function public.handle_new_user()
       returns trigger
       language plpgsql
       security definer
       set search_path = public
       as $$
       begin
         insert into public.profiles (id, email, full_name, avatar_url)
         values (
           new.id,
           new.email,
           coalesce(new.raw_user_meta_data->>'full_name', ''),
           coalesce(new.raw_user_meta_data->>'avatar_url', '')
         )
         on conflict (id) do nothing;
         return new;
       end;
       $$;

       create trigger on_auth_user_created
         after insert on auth.users
         for each row execute function public.handle_new_user();
       ```

       Note: `security definer` is required because the trigger runs in auth system context. `set search_path = public` prevents search path injection — without it, a malicious user could create a function named like a system function and hijack execution.

    Avoid: Adding RLS policies (plan 02-06). Business logic triggers (those belong in feature phases). Triggers referencing tables not yet created.
