<task type="auto">
  <name>Task 1: Fleet Schema Migration</name>
  <files>supabase/migrations/20260411130000_fleet_schema.sql</files>
  <action>
    Create fleet schema migration with 3 tables. Follow exact patterns from 02-01 core schema (column naming, comments, index naming, migration header format).

    **vehicle_types table:**
    - id: uuid, gen_random_uuid(), PK
    - name: text NOT NULL UNIQUE (e.g., "Giuong nam 40 cho")
    - description: text
    - seat_layout: jsonb NOT NULL CHECK (jsonb_typeof(seat_layout) = 'object') — JSON schema for seat map (floors, rows, seat positions) <!-- audit-added -->
    - total_floors: int NOT NULL DEFAULT 1 CHECK (total_floors > 0)
    - total_seats: int NOT NULL CHECK (total_seats > 0)
    - amenities: jsonb NOT NULL DEFAULT '[]'::jsonb CHECK (jsonb_typeof(amenities) = 'array')
    - created_at: timestamptz NOT NULL DEFAULT now()
    - updated_at: timestamptz NOT NULL DEFAULT now()
    - Indexes: GIN on seat_layout, GIN on amenities
    - Comments on table and all columns

    **vehicles table:**
    - id: uuid, gen_random_uuid(), PK
    - vehicle_type_id: uuid NOT NULL FK -> vehicle_types(id) ON DELETE RESTRICT
    - license_plate: text NOT NULL UNIQUE (Vietnamese format: 51A-12345)
    - vin_number: text UNIQUE (vehicle identification number, nullable)
    - year_manufactured: int CHECK (year_manufactured >= 1990 AND year_manufactured <= EXTRACT(YEAR FROM now()) + 1)
    - status: text NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'maintenance', 'retired'))
    - current_mileage: int DEFAULT 0 CHECK (current_mileage >= 0)
    - last_maintenance_date: date
    - next_maintenance_date: date
    - notes: text
    - created_at: timestamptz NOT NULL DEFAULT now()
    - updated_at: timestamptz NOT NULL DEFAULT now()
    - Indexes: idx on vehicle_type_id, idx on status, idx on next_maintenance_date
    - Comments on table and all columns

    **maintenance_logs table:**
    - id: uuid, gen_random_uuid(), PK
    - vehicle_id: uuid NOT NULL FK -> vehicles(id) ON DELETE CASCADE
    - type: text NOT NULL CHECK (type IN ('routine', 'repair', 'inspection', 'emergency'))
    - description: text NOT NULL
    - cost: numeric(12,2) DEFAULT 0 CHECK (cost >= 0)
    - performed_by: text
    - performed_at: date NOT NULL DEFAULT CURRENT_DATE
    - next_due_date: date
    - odometer_reading: int CHECK (odometer_reading >= 0)
    - notes: text
    - created_at: timestamptz NOT NULL DEFAULT now()
    - updated_at: timestamptz NOT NULL DEFAULT now()
    - Indexes: idx on vehicle_id, idx on type, idx on performed_at, idx on next_due_date
    - Comments on table and all columns

    Include migration header with version, description, prerequisites (requires 20260410120000_core_schema.sql).

    Avoid: Adding any RLS policies (deferred to 02-06), business logic triggers beyond updated_at (deferred to 02-07), or detailed seat layout visualization JSON (deferred to Phase 3).
  </action>

<task type="auto">
  <name>Task 3: Fleet Seed Data</name>
  <files>supabase/seed.sql</files>
  <action>
    APPEND fleet seed data to existing seed.sql. Preserve all existing core seed data (roles, employees, user_roles). Add clear section separator comment.

    **Vehicle Types (5 records) — use CTE to capture IDs:**

    1. "Giuong nam 40 cho" — 40-bed sleeper, 2 floors, 40 seats, amenities: ["wifi","charging","ac","blanket"], seat_layout: basic 2-floor grid structure
    2. "Limousine 24 cho" — 24-seat limousine, 1 floor, 24 seats, amenities: ["wifi","charging","ac","tv","water"], seat_layout: 3-seat-per-row grid
    3. "Ghe ngoi 29 cho" — 29-seat standard, 1 floor, 29 seats, amenities: ["ac","charging"], seat_layout: 2+2 bus layout
    4. "Giuong nam VIP 22 cho" — 22-bed VIP, 2 floors, 22 seats, amenities: ["wifi","charging","ac","tv","blanket","curtain"], seat_layout: 2-floor grid
    5. "Cabin 11 cho" — 11-seat cabin, 1 floor, 11 seats, amenities: ["wifi","charging","ac","tv","water","snack"], seat_layout: single row layout

    seat_layout JSONB structure (functional, not final — refined in Phase 3):
    ```json
    {"floors": N, "rows": R, "cols_per_row": C, "seat_ids": ["A1","A2",...]}
    ```

    **Vehicles (8 records) — reference vehicle_type IDs via CTE:**
    - 2x Giuong nam 40 (51A-12345 active, 51A-12346 active)
    - 2x Limousine 24 (51B-67890 active, 51B-67891 active)
    - 2x Ghe ngoi 29 (51C-11111 active, 51C-11112 maintenance)
    - 1x Giuong nam VIP 22 (51D-22222 active)
    - 1x Cabin 11 (51E-33333 retired)
    Mix: year_manufactured 2019-2024, current_mileage 50,000-320,000, various last/next maintenance dates.

    **Maintenance Logs (6 records) — reference vehicle IDs via CTE:**
    - 2 routine, 2 repair, 1 inspection, 1 emergency
    - Costs in VND: 500,000 to 15,000,000
    - Dates within past 6 months
    - Various performed_by mechanic names

    All inserts use ON CONFLICT DO NOTHING for idempotency. <!-- audit-added: explicit targets below -->
    Use explicit conflict targets matching UNIQUE constraints:
    - vehicle_types: ON CONFLICT (name) DO NOTHING
    - vehicles: ON CONFLICT (license_plate) DO NOTHING
    - maintenance_logs: ON CONFLICT (id) DO NOTHING

    Use SELECT-based FK resolution instead of CTE INSERT...RETURNING, because RETURNING returns no rows when ON CONFLICT DO NOTHING triggers. Pattern: <!-- audit-added -->
    ```sql
    -- Insert vehicle types first
    INSERT INTO vehicle_types (...) VALUES (...) ON CONFLICT (name) DO NOTHING;

    -- Reference vehicle types by name for vehicle inserts
    INSERT INTO vehicles (vehicle_type_id, license_plate, ...)
    SELECT vt.id, '51A-12345', ...
    FROM vehicle_types vt WHERE vt.name = 'Giuong nam 40 cho'
    ON CONFLICT (license_plate) DO NOTHING;

    -- Reference vehicles by license plate for maintenance log inserts
    INSERT INTO maintenance_logs (vehicle_id, type, ...)
    SELECT v.id, 'routine', ...
    FROM vehicles v WHERE v.license_plate = '51A-12345'
    ON CONFLICT (id) DO NOTHING;
    ```
