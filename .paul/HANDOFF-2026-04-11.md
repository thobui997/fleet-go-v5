## What Was Done This Session

**Plan 02-01: Core Schema** — Complete
- Created 4 core tables: `profiles`, `roles`, `user_roles`, `employees`
- Normalized schema design: profiles as single source of truth for user data
- Composite PK on `user_roles(user_id, role_id)` — no surrogate key
- `employees.user_id` UNIQUE nullable — enforces 1:1 with edge case support
- JSONB permissions with GIN index — dynamic RBAC without schema changes
- Secure triggers: `handle_new_user()` with `SECURITY DEFINER` + `SET search_path = public`
- Auto-`updated_at` trigger on all core tables
- Vietnamese seed data: 9 users, 9 employees, 6 roles, 9 role assignments

**Deviation from plan:**
- Original plan: Direct `auth.users` insertion using pgcrypto extension
- Actual: Dashboard user creation + employee/role seeding
- Reason: pgcrypto's `gen_salt()` function not available in Supabase Dashboard SQL Editor
- Impact: Additional manual step but more reliable; documented in seed.sql

## Key Decisions Made

| Decision | Rationale |
|----------|-----------|
| Normalized user schema | profiles as single source of truth eliminates duplication |
| Composite PK on junction tables | Eliminates redundant surrogate keys, enforces uniqueness naturally |
| JSONB permissions with GIN index | Dynamic permissions without schema changes, fast @> queries |
| Dashboard user creation for seed | pgcrypto extension unreliable in Supabase Dashboard SQL Editor |
| UNIQUE nullable on employees.user_id | Enforces 1:1 relationship but allows multiple NULLs for edge cases |
